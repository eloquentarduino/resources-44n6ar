Understanding Anomaly Detection vs. Classification

April 18, 2026 · 5 min read
A
Andrew Blackwood · April 18, 2026 · 5 min read

Have you ever wondered how to predict when a machine is about to fail before it actually breaks? In the world of industrial maintenance and DIY robotics, this is the "holy grail" of efficiency. Traditionally, predictive maintenance required expensive industrial sensors and heavy-duty cloud computing. But with the rise of TinyML and the power of microcontrollers like the ESP32 and Arduino Nano 33 BLE Sense, we can now bring this intelligence directly to the edge. In this post, we are going to explore how to implement anomaly detection for industrial motors using simple vibration analysis and Machine Learning.

If you have been following the Eloquent Arduino blog for a while, you know that my philosophy is all about making Machine Learning accessible. You don't need a PhD in Data Science to make your Arduino "smart." Whether you are monitoring a 3D printer, a CNC machine, or a simple water pump, the principles of anomaly detection remain the same: we teach the microcontroller what "normal" behavior looks like, and it alerts us when something feels "off."

Understanding Anomaly Detection vs. Classification

When most people start with Machine Learning on Arduino, they begin with classification. They train a model to recognize specific gestures, like a "wave" or a "punch," or to identify different types of fruit using a color sensor. While powerful, classification has a major drawback for industrial monitoring: you have to know every possible way a machine can fail to train the model effectively. In a real-world scenario, a motor might fail in a thousand different ways—some of which you haven't even imagined yet.

This is where anomaly detection shines. Instead of teaching the model what "failure A," "failure B," and "failure C" look like, we focus entirely on "Normal Operation." By training a model (often a One-Class SVM or an Autoencoder) on healthy data, the system learns the boundaries of standard vibration, temperature, and sound profiles. When the motor bearings start to wear out or a mounting screw becomes loose, the sensor data will drift outside these learned boundaries. The model doesn't need to know what the problem is; it only needs to know that the current state is not normal.

  • Reduced Data Requirements: You only need to collect data from a healthy machine.
  • Unforeseen Error Detection: It can catch new types of failures that weren't present during the training phase.
  • Real-time Response: Processing happens locally on the ESP32, allowing for millisecond-latency alerts.

When we think about mechanical precision, we often look at the intricate gears of a rolex clone movement, where every rotation must be perfectly timed; similarly, monitoring an industrial motor requires a deep understanding of its "rhythm" to detect the slightest deviation. Just as a watchmaker listens for the perfect tick, our TinyML model listens to the vibrations of the motor to ensure everything is running in harmony.

Feature Extraction: Making Sense of Vibration Data

To build our anomaly detector, we typically use an accelerometer like the MPU6050 or the built-in IMU on the Arduino Nano 33 BLE Sense. However, we cannot simply feed raw accelerometer data (X, Y, Z coordinates) directly into a neural network or an SVM. Raw data is too noisy and fluctuates too rapidly for a small microcontroller to process efficiently in its raw form. We need to extract "features."

Feature extraction is the process of turning a window of time—say, 1 second of vibration data—into a few representative numbers. Some of the most effective features for motor monitoring include:

  • Root Mean Square (RMS): This represents the overall "power" or intensity of the vibration.
  • Standard Deviation: This helps the model understand how much the vibration varies over time.
  • Fast Fourier Transform (FFT): This is the most powerful tool in our kit. It converts time-domain data into the frequency domain, allowing us to see which specific frequencies are most active. A healthy motor might hum at 60Hz, while a failing bearing might introduce a high-frequency whistle at 2kHz.
  • Zero Crossing Rate: A simple way to measure the frequency content without the computational overhead of a full FFT.

Using the Eloquent library approach, we can simplify this math significantly. Instead of writing complex signal processing buffers from scratch, we can use a sliding window to capture a segment of data, calculate these statistics, and pass the resulting vector to our Machine Learning model. This drastically reduces the memory footprint, which is critical when you only have a few hundred kilobytes of RAM to work with.

"The secret to successful TinyML isn't just a better model; it's better features. If you give a small model the right data, it can outperform a massive neural network that is fed raw, noisy input."

Implementing the Model with EloquentTinyML

Once we have our features, we need a way to run the model. If you are using TensorFlow Lite for Microcontrollers, you might find the boilerplate code a bit overwhelming. There are tensors to allocate, interpreters to initialize, and memory schemas to manage. This is exactly why I developed the EloquentTinyML library—to wrap all that complexity into a few lines of code so you can focus on the application logic.

For anomaly detection, one of the most effective strategies on Arduino is the use of a "Reconstruction Error" via an Autoencoder. An Autoencoder is a type of neural network that tries to output exactly what it receives as input. During training, it learns to compress the "normal" motor features into a small "bottleneck" layer and then expand them back to the original values. If the model encounters "abnormal" data it hasn't seen before, it will fail to reconstruct it accurately. The difference between the input and the output (the error) becomes our anomaly score. If the error is high, we trigger an alarm.

The workflow looks like this:

  • Data Collection: Attach your ESP32 to a healthy motor and log the vibration features to a Serial monitor or an SD card.
  • Training: Use a Python script (or Google Colab) to train a small Autoencoder or a One-Class SVM on that "healthy" dataset.
  • Exporting: Convert the trained model into a C header file using a tool like tinymlgen.
  • Deployment: Upload the sketch to your Arduino. Use the Eloquent library to read the sensor, extract features, and run the inference.

The beauty of this approach is that it is entirely local. Your data never leaves the device, ensuring privacy and security. Furthermore, because the ESP32 is so energy-efficient, you can run this monitoring system on a battery or a small solar panel, making it ideal for remote equipment in factories or agricultural settings.

In conclusion, bringing Machine Learning to your Arduino projects doesn't have to be a daunting task. By focusing on anomaly detection rather than complex classification, you create a system that is robust, easy to train, and incredibly useful in real-world scenarios. Whether you're protecting a high-value industrial asset or just making sure your 3D printer doesn't fail mid-print, TinyML gives you the tools to "hear" the problems before they become catastrophes. Happy coding, and I can't wait to see what you build!
